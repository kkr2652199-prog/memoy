### B안) 가중치 자동 반영 — **현재 구조 유지** + “이미 DB를 쓰는가?” **사실 확인**

| 항목 | 내용 |
|------|------|
| **사실(코드)** | `_v12_merged_hyena_weights`(**`v12_engine.py` 54~64행**): **`v12_stat`·`v12_run`·`v12_offset`·`v12_contrarian`·`v12_lstm`·`v12_hyena` 키는 `get_v12_brain_weights()`로 DB 반영**. 단, **`v12_fusion`은 매번 `2.0`으로 강제**되어 **Hedge 테이블의 고가중치(예: 125)는 hyena 합의에 반영되지 않음**. |
| **장점** | **변경 없음**으로도 **5개 기저 뇌**는 이미 **진화 가중치**를 탄다. 1군 `_load_hyena_brain_weights`도 fusion만 FALLBACK(**`predict_hyena.py` 34~38행**)이라 **패턴 일관**. |
| **단점** | “DB 자동 반영 = fusion까지”라는 **기대와 불일치**가 있을 수 있음. |
| **코드 변경 범위** | **0**(현상 유지) ~ **소규모**(fusion도 DB 쓰도록 **63행 제거/조건부화**) — 후자는 **1군과의 미러 정합**·**백테 재검증** 필요. |
| **리스크** | fusion 가중을 DB에 맡기면 **5005 합의 점수 스케일**이 변해 **tier1·샘플링 분포**가 크게 흔들릴 수 있음. |

### C안) 1군 미러링 — 입력을 **1군과 동일 5뇌(stat, markov, llm, lstm, fusion)**에 맞추되 **`llm` 대신 `v12_snake` 투입**

| 항목 | 내용 |
|------|------|
| **장점** | **태그 네임스페이스**를 1군과 맞추면 **문서·디버깅·비교**가 쉬움. `v12_snake`는 백테에서 **평균 적중이 높은 편**(0200 §3-C)이라 **“강한 대체 뇌”** 역할 기대 가능. |
| **단점** | **현재 파이프라인 순서**상 `v12_snake`는 **`_v12_hyena_predict` 이후**에 생성됨(**`v12_engine.py` 307 vs 312행**). **그대로는 snake를 hyena 입력에 넣을 수 없음** — **순서 뒤바꿈·2패스·snake 간이 프리패스** 등 **아키텍처 변경** 필요. |
| **코드 변경 범위** | **대규모**: (1) snake 이전에 **저비용 대리 입력**을 만들거나 (2) hyena를 **snake 뒤로 이동**하면 **DB INSERT·가중치 갱신 순서**·**캐시 40행** 조건까지 연쇄 검토. `_normalize_brain_tag`/`가중치 키` 매핑 추가. |
| **리스크** | **순환 의존**: snake가 이미 hyena·타 뇌 출력을 쓰는 설계라면 **이중 카운팅**·**합의 왜곡**. **R29** 1·2군 READ-ONLY와의 경계도 재확인. |

### 5-C) 관찰(단정 아님)

- 동일 당첨·**다른 기저 입력·다른 가중치·다른 fusion 고정(3군)·다른 v12_fusion_v5**가 동시에 작용하므로, **「입력만 바꾸면 1군과 동일 6적중」**으로 연결하긴 어렵다.  
- **1037**: 1군·3군 모두 **당첨 6수와 높은 교집합** 세트가 존재하나, **3군 hyena 최고는 5**로 **한 번호(예: 22 vs 18)** 수준에서 엇갈림.
