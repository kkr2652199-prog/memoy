- **인접 행렬**: `lotto_cooccur_3` 전체 행에서 트리플의 세 페어에 `count` 가중치 합산(대칭). `lotto_cooccur_4` 전체 행에서 4개 번호의 6페어에 `count * 0.5` 합산.

- **클러스터 선택**: `node_score` 상위 **20**개 풀에서 첫 풀은 `node_score` 비례 샘플링, 이후 `pick = node_score[n] + 0.3 * mean(adj[selected, n])`로 가중 샘플(`numpy.random.Generator`, 시드는 `draw_no` 기반).
- **필터**: 클러스터 후보는 합계 100~175, 홀수 개수 2~4, 자카드 0.5 미만. 부족 시 `generate_sets_with_filters(score_dict, …)`로 보충(기본 홀짝 규칙). 데이터 없으면 랜덤 세트.

## 후속 제안

1. **성능**: 회차마다 `cooccur_3/4` 전체 스캔은 비용이 큼. DB 변경이 없다면 `(db_path, mtime)` 기준 **정적 인접(3+4) 캐시**를 두고, 요청마다 **recency + draws**만 가산하는 방식으로 최적화 가능.
2. **PageRank**: 2회는 경량용; 반복 수·댐핑은 오프라인 백테스트로 조정 여지 있음.
3. **풀 크기**: `CANDIDATE_POOL=20` 고정 대신 차수 급이 큰 구간에서만 풀을 넓히는 적응형도 검토 가능.
4. **앙상블**: `ensemble`에 그래프 출력이 합류했으므로, 가중치 학습 시 graph 태그 누적 통계를 별도 모니터링하는 것이 좋음.
