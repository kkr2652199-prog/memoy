## 파트 1 — 백테스트 무누수(컨닝) 검증 ★

### 1) target 이전 draws만 예측 생성에 쓰이는가?

**예 — 당첨번호 입력(draws) 컷오프는 정직.**

| 근거 | 파일:줄 |
|------|---------|
| `_get_draws_before`: `WHERE draw_no < ?` — target **미포함** | `data_service.py:896-904` |
| `run_prediction` 신규 생성: `draws = _get_draws_before(target_draw_no)` 후 5뇌 호출 | `engine.py:289-341` |
| stat/markov/llm/lstm/fusion 모두 `draws` 인자만 사용 | `engine.py:306-337`, `predict_llm.py:28-31` |
| target 당첨번호가 **생성**에 쓰이는 경로 | **없음** (코드 전수 확인) |

target 당첨번호는 **INSERT 직전 채점**에만 사용:

| 단계 | 파일:줄 |
|------|---------|
| `actual_row = SELECT … WHERE draw_no = target` | `engine.py:428-443` |
| `matched_count`·`bonus_matched` 계산 후 INSERT | `engine.py:445-472` |

→ 번호 생성과 정답 조회는 **같은 함수 내 순차**이나, 생성은 이미 `draws`(target 미포함)로 끝난 뒤.

### 2) 채점(scoring) vs 예측(generation) 시간 분리?

**예측 저장(INSERT) 후·동일 트랜잭션 직전에 채점 — 생성은 항상 선행.**

| 흐름 | 파일:줄 |
|------|---------|
| 5뇌 번호 생성 → `to_insert` | `engine.py:305-419` |
| 당첨 조회 → matched 계산 → INSERT | `engine.py:428-472` |
| 백테스트: `run_prediction` → DB에서 matched 조회 → 피드백 | `engine.py:579-609` |
| 사후 채점 훅(당첨 확정 후) | `refresh_prediction_scores_for_target_draw` `engine.py:65-88` |

백테스트 루프 순서: **예측 → (INSERT 시 채점) → 피드백 → 가중치 갱신** (`engine.py:579-609`).

### 3) brain_weights / feedback가 target 이후 데이터로 target 예측에 역류?

**부분 역류 가능 — fusion·LLM·마르코프 보조 경로.**

| 항목 | walk-forward 의도 | 실측 리스크 | 근거 |

| `_load_brain_weights_from_db()` | target별 컷오ff **없음** | DB `last_updated_draw=1232` 상태에서 1100회 백테스트 시 **미래 가중치**로 fusion | `fusion.py:220`, `feedback.py:546-568` — 실측: 5행 모두 `last_updated_draw=1232` |

### 파트 1 결론 (1줄)

**과거 백테스트는 [부분누수] — draws 컷오프·INSERT 채점 분리는 정직하나, fusion DB 가중치·LLM/마르코프 feedback 요약·예측 캐시가 target 무관 전역 상태를 쓰면 미래 정보·구예측이 스며들 수 있음.**

## 복제 vs 수정 — 커서 소견 (READ-ONLY 관찰)

1. **draws 컷오프·채점 분리**는 설계 의도대로 구현돼 있어, “번호 생성에 target 당첨번호 넣는” **직접 컨닝은 없음**.
2. **fusion DB 가중치·feedback 요약·예측 캐시**는 target 무관 전역 상태라, 현 DB(`last_updated_draw=1232`) 기준 **과거 구간 백테스트 재실행 시 부분누수**가 현실적이다 — 복제 전에 “WF 전용 가중치 스냅샷” 또는 컷오ff 로직 없으면 수치 신뢰도는 낮다.
3. **LLM 홀딩·LSTM uniform fallback**으로 7뇌 중 실질 엔진은 stat/markov/fusion/hyena/lead1 쪽에 치우쳐 있고, UI는 7뇌 풀세트처럼 보인다.
4. **규모**: lotto Python ~6.6k줄 + UI ~3k줄 + DB ~50MB×2 — “전면 복제”보다 **1군 디렉터리+DB+모델만 스냅샷**하고, WF·가중치·캐시 정책을 **현 구조 수정**으로 고치는 쪽이 diff가 작을 가능성이 크다.
5. **postmortem/pattern_store**는 API·UI 없이 DB·훅만 있어, 복제 시 “죽은 코드”로 오해하기 쉽다 — 정리 후보는 문서화만 하고 이번 READ-ONLY에서는 미삭제.
