## STEP 7 — A안 vs B안 영향 범위

| 항목 | A안 (키 추가: `v12_run`을 hyena dict·CONTRIBUTING에 추가) | B안 (tag 호환: run 출력 `brain_tag`를 `v12_stat`로 유지) |
|------|----------------------------------------------------------|----------------------------------------------------------|
| 수정 파일 수 | 주로 **`v12_engine.py` 1개** (가중치·frozenset; stat 슬롯 호출 교체는 별도) | 주로 **`v12_engine.py` + 신규 `predict_run`의 반환 필드** (엔진은 2줄 근처 가능) |
| 수정 라인 수 | **소수 줄** (키 2곳 + 필요 시 주석) | **적음** (단, DB/API에 **stat vs run 구분 불가**) |
| hyena 영향 | **의도 명확**: `v12_run` 세트도 `weights.get`에 걸림 | **안전**: 기존 `v12_stat` 키와 100% 호환 |
| 1·2군 영향 | **0** (본 정찰 범위에서 `lotto`/`lotto2` 미수정 전제) | **0** |
| 롤백 난이도 | 낮음 (git/백업으로 dict 2블록 되돌리기) | 낮음 |
| 디버깅 난이도 | 낮음·중간 — DB에 `v12_run` 행 존재, 대시보드에 별도 행으로 잡힐 수 있음 | **높음** — 같은 태그로 묶여 run 전용 집계·UI 구분 어려움 |

**판단 근거 (정찰 기반):**

- hyena는 **변수 `stat`가 아니라 `fresh[]`의 `brain_tag`**로만 합의한다.
- 따라서 **`brain_tag == "v12_run"`인데 가중치 dict에 `v12_run`이 없으면 가중치 0** → A안 없이 run만 넣으면 **합의에서 사실상 제외**.
- B안은 hyena·CONTRIBUTING·DB 가중치 테이블(`v12_models`의 `V11_BRAINS`가 `v12_stat`만 추적)과 **충돌이 적음** 대신 **의미상 stat/run 혼선**이 생김.

**권장:** **운영·분석 분리가 목적이면 A안**, **최소 줄수·즉시 호환만이면 B안** — 단 B안은 “첫 독창 뇌”를 DB에서 식별하기 어렵다는 트레이드오프가 있음.
